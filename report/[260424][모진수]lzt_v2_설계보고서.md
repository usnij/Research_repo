## 1. LZT의 한계: Alpha 비보존 문제

LZT(Linear Z-Thickness)는 SFM(Smooth Fragment Merging) 방식으로 overlap을 처리한다:

```
G1+G2 overlap → zone1(G1만) + zone2(G1 back + G2 front) + zone3(G2만)
각 zone을 독립 over-operator로 compositing
```

이때 각 zone에서의 alpha는:
```
zone2 기여:  A_F2 = r_F2 * A_F,  A_B2 = r_B2 * A_B

zone2 합성:  A_zone2 = A_F2 + A_B2 * (1 - A_F2)
```

**문제**: zone 분할 후 각 Gaussian의 실제 compositing alpha가 달라진다.
- G1의 총 기여: `A_F1 + A_zone2 * (1 - A_F1) ≠ A_F` (일반적으로)
- G2도 마찬가지

**결과**: 학습 중 각 Gaussian은 자신의 원래 alpha를 보존하도록 최적화되지만, SFM forward는 이를 변형해서 사용한다 → forward-backward 불일치 → baseline 미달.

## 2. lzt_v2의 핵심 아이디어

### 2.1 Linear Alpha 근사

각 Gaussian의 alpha가 깊이 구간에 **선형 비례**한다고 근사:

```
α_zone = alpha_G × len_zone / hlen_G
```

여기서:
- `alpha_G` = Gaussian의 2D projected alpha (원래 값)
- `len_zone` = zone의 깊이 길이
- `hlen_G = t2_G - t1_G` = Gaussian 전체 깊이 구간 길이

이 근사 덕분에 zone alpha들의 합이 **자동으로** alpha_G를 보존:

```
Σ α_zone_i = alpha_G × Σ len_zone_i / hlen_G = alpha_G  ✓
```

### 2.2 단일 Mix Operator

zone2 혼합(여러 Gaussian의 겹침)과 zone 병합 모두 **동일한 weighted average**:

```
mix({(α_i, c_i)}) = Σ(α_i × c_i) / Σ(α_i)
```

### 2.3 Alpha 보존 증명

```
G1:  α_G1_z1 + α_G1_z2
   = alpha_G1 × len_z1 / hlen_G1 + alpha_G1 × len_z2 / hlen_G1
   = alpha_G1 × (len_z1 + len_z2) / hlen_G1
   = alpha_G1 × hlen_G1 / hlen_G1
   = alpha_G1  ✓

G2:  α_G2_z2 + α_G2_z3
   = alpha_G2 × (len_z2 + len_z3) / hlen_G2
   = alpha_G2  ✓
```

각 Gaussian의 compositing alpha = 원래 alpha_G → **변하지 않음**.

## 3. Zone 구조 (2-way Overlap)

```
G1: [t1_G1 ──────────── t2_G1]
G2:        [t1_G2 ──────────────── t2_G2]

     [  zone1  ][    zone2    ][    zone3    ]
         G1         G1 + G2          G2
```

| Zone | 범위 | 포함 Gaussian | α_G1 기여 | α_G2 기여 |
|------|------|--------------|-----------|-----------|
| zone1 | [t1_G1, t1_G2] | G1만 | `alpha_G1 × len_z1 / hlen_G1` | — |
| zone2 | [t1_G2, t2_G1] | G1 + G2 | `alpha_G1 × len_z2 / hlen_G1` | `alpha_G2 × len_z2 / hlen_G2` |
| zone3 | [t2_G1, t2_G2] | G2만 | — | `alpha_G2 × len_z3 / hlen_G2` |

**핵심 차이 (vs LZT/GFB)**: zone2는 G1과 G2가 **공유하는 mixed entity**다.
- G1 drain 시: zone1 + zone2를 merge해 G1의 c_eff 계산 → G1 composite
- G1 drain 후: **zone2는 제거하지 않음** → G2 drain 시 재사용
- G2 drain 시: zone2 + zone3을 merge해 G2의 c_eff 계산 → G2 composite

## 4. Drain 알고리즘

k-buffer는 t2 기준으로 정렬하며, t2가 가장 작은 Gaussian(G_front)을 순서대로 drain한다.

---

### 4.1 Zone 분할

두 Gaussian이 겹치면 깊이 구간이 세 zone으로 분할된다:

```
G1:  [t1_G1 ──────────── t2_G1]
G2:            [t1_G2 ──────────────── t2_G2]

     ┌─ zone1 ─┬──── zone2 ────┬──── zone3 ────┐
     │  G1 only│  G1+G2  혼합  │    G2 only    │
     └─────────┴───────────────┴───────────────┘
```

zone1, zone3은 한 Gaussian만 존재하는 단독 구간이고,  
**zone2는 G1과 G2가 함께 기여하는 혼합 구간**이다.

---

### 4.2 Zone2 혼합 방식

각 Gaussian이 zone에 기여하는 alpha는 **전체 깊이 구간 대비 zone 길이**에 선형 비례한다:

```
α_G1→z2 = alpha_G1 × len_z2 / hlen_G1        (hlen = t2 − t1)
α_G2→z2 = alpha_G2 × len_z2 / hlen_G2
```

zone2의 혼합 색상은 이 alpha로 가중 평균한다:

```
c_zone2 = (α_G1→z2 × c_G1  +  α_G2→z2 × c_G2)
        / (α_G1→z2 + α_G2→z2)
```

이 정의에 의해 각 Gaussian의 alpha가 자동으로 보존된다:

```
α_G1→z1 + α_G1→z2  =  alpha_G1 × (len_z1 + len_z2) / hlen_G1  =  alpha_G1  ✓
α_G2→z2 + α_G2→z3  =  alpha_G2  ✓
```

---

### 4.3 Drain: 소속 Zone 병합 → 유효 색상 계산

G_front를 drain할 때, **G_front에 속한 모든 zone을 병합**하여 유효 색상 c_eff를 구한다.

**G1 drain** — zone1(단독) + zone2(혼합) 병합:

```
c_eff_G1 = (α_G1→z1 × c_zone1  +  α_G1→z2 × c_zone2) / alpha_G1

color += T × alpha_G1 × c_eff_G1
T'     = T × (1 − alpha_G1)
```

**zone2는 G1 drain 후에도 유지된다.** G2가 drain될 때 zone2를 재사용한다.

**G2 drain** — zone2(혼합, 재사용) + zone3(단독) 병합:

```
c_eff_G2 = (α_G2→z2 × c_zone2  +  α_G2→z3 × c_zone3) / alpha_G2

color += T × alpha_G2 × c_eff_G2
T'     = T × (1 − alpha_G2)
```

---

### 4.4 파트너 캐싱

G1이 drain되면 k-buffer에서 사라지지만, G2의 zone2 재계산 시 G1 데이터가 필요하다.  
→ G1 drain 시점에 G1 데이터를 G2의 `cachedPartners`에 저장해두고, G2 drain 시 복원한다.

```
G1 drain → G2.cachedPartners ← {G1.alpha, G1.t1, G1.t2, c_G1}
G2 drain → cachedPartners에서 G1 복원 → c_zone2 재계산
```

## 5. N-way Overlap 확장: composited_set

3-way 이상 overlap 시 문제:

```
buffer = [zone1(G1+G2), zone2(G1+G2+G3), ...]

G1 drain 후 → zone2에서 min-t2 탐색 시 여전히 G1 → 무한루프
```

**해결**: drain 루프 전체에 걸쳐 **composited_set** (이미 composite된 Gaussian 집합) 유지.

구현:
1. drain 대상 선택 시: 현재 맨 앞 zone의 Gaussian 중 composited_set 제외 후 t2 최소 선택
2. mix 계산 시: 각 zone에서 composited_set과 무관하게 **현재 살아있는 파트너** 기준으로 계산

실제 구현에서는 `composited_set` 대신 **CachedPartner 메커니즘**으로 이를 자연스럽게 처리한다.

## 6. CUDA 구현 설계

### 6.1 CachedPartner 메커니즘

drain 후 k-buffer에서 제거된 G_front의 데이터를 보존하는 구조:

```cpp
struct HitParticle {
    // 기존 필드
    int idx; float hitT, alpha, t1, t2, grduLen, disc;

    // 파트너 캐시
#if GAUSSIAN_ZPGC
    struct CachedPartner {
        int   idx   = -1;
        float alpha = 0.f;
        float t1    = 0.f;
        float t2    = 0.f;
    };
    static constexpr int MaxCachedPartners = GAUSSIAN_ZPGC_MAX_PARTNERS;
    CachedPartner cachedPartners[MaxCachedPartners > 0 ? MaxCachedPartners : 1];
    int numCachedPartners = 0;
#endif
};
```

G_i drain 시 zone mix 계산:
- 아직 k-buffer에 있는 파트너 → k-buffer에서 직접 조회 (kbufSlot >= 0)
- 이미 drain된 파트너 → cachedPartners에서 조회 (kbufSlot = -1)

### 6.2 레지스터 예산

```
HitParticle 1개 = 7(기본 필드) + MAX_PARTNERS×4(파트너) + 1(numCached) registers
총 k-buffer = K × HitParticle + ~50(기타 오버헤드)
```

| K | MAX_PARTNERS | 총 registers | occupancy | 판정 |
|---|-------------|-------------|-----------|------|
| 10 | 2 | ~210 | ~50% | ✓ 기본 운용 |
| 10 | 3 | ~250 | ~25% | △ 연구 목적 허용 |
| 12 | 2 | ~230 | ~33% | △ 빡빡하나 가능 |
| 12 | 3 | ~278 | 한도 초과 | ✗ |
| 16 | any | — | 별도 최적화 필요 | 교수님 논의 |

### 6.3 결정사항

1. **학습 기준 K = 10~12** (레지스터 압박 허용 범위)
2. **MAX_PARTNERS = 2** 기본값 (K=10 ~50% occupancy)
3. **MAX_PARTNERS = 3** 은 K=10에서만 가능 (연구 목적)
4. **CompactShift template parameter**: mid-traversal = `false` (invalidateFront), final drain = `true` (drainFront)
5. 설정 파라미터: `zpgc: bool`, `zpgc_max_partners: int`

## 7. Backward 설계

### 원칙: G_front 독립 처리

lzt_v2 forward에서 G_front의 c_eff는 파트너들의 color와 zone mix로 결정된다. 정확한 backward는 파트너들에게도 gradient를 전파해야 하지만, 이는 GFB에서 확인한 것처럼 3DGUT 학습 방향과 충돌한다.

**채택한 근사**: **G_front의 gradient만 계산**, 파트너들에게는 gradient를 전파하지 않는다.

```
G_front backward:
  c_eff 재계산 (forward와 동일한 zone mix)
  → c_eff를 G_front의 색상으로 취급
  → featuresIntegrateBwd + densityProcessHitBwdToBuffer 호출

파트너들: G_front drain 단계에서 NO gradient
  (각 파트너는 자신의 drain 단계에서만 gradient 수신)
```

각 Gaussian의 gradient는 **자신이 G_front가 되는 drain 단계에서만** 계산된다. 이 방식은:
- zone 비율의 t1/t2 geometric gradient를 완전히 배제
- GFB 실패 원인(geometric gradient와 학습 방향 충돌)을 회피
- LZT backward의 "독립 처리" 철학과 동일
